# **개요**

- 차량 내에서 **호스트 컴퓨터 **없이 마이크로 컨트롤러나 장치들이 통신하기 위해 설계된 **표준 통신 규격**
- non-host Bus 방식의 메시지 기반 네트워크 프로토콜
- 모든 노드들은 물리적으로 two-wire구조의 버스 형태로 이어져 있다.
- 두 가지 시그널이 존재 - CAN HIGH / CAN LOW
- 0 bit → 우성 / 1 bit → 열성 => ID가 낮을수록 우선순위가 높아짐

## 노드

![CAN Bus 노드 구조](/images/notion/can-bus/image-1.png)

각 노드는 아래와 같은 요소를 포함한다.

- Central Processing Unit(or microprocessor, host processor)
	- 전송된 메시지가 무엇인지, 어떤 메시지를 전송할지 결정한다.
	- 센서, 액추에이터, 그리고 컨트롤 장치가 연결될 수 있다.
- CAN controller(microcontroller의 일부분이 될 수 있다)
	- Receiving
		- 전체 메시지가 유효해질 때까지 bus로부터 일련(serial)의 bits를 받고 저장한다.
		- host processor가 해당 메시지를 가져올 수 있음
	- Sending
		- Bus가 비어있을 때, host processor가 CAN controller에게 버스에 직렬로 비트를 태워 보내서 전송 메시지(transmit messages)를 보낸다.
- CAN Transceiver
	- Receiving
		- CAN bus에서 받은 데이터 스트림을 변환하여 CAN controller가 사용할 수 있도록 한다.
	- Transmitting
		- CAN controller로부터 데이터 스트림을 변환하여 CAN bus에게 넘겨준다.

각 노드는 메시지를 전송하고 받을 수 있지만, **동시**에 하지는 못한다.
메시지 또는 프레임은 주로 메시지의 우선순위를 나타내는 최대 8bytes의 ID로 구성된다.
CRC, ACK 및 기타 오버헤드도 메시지의 일부이다.

**CAN FD**는 데이터 섹션의 길이를 최대 64바이트까지 확장한다.

메시지는 NRZ(Non-return-to-zero)형식을 사용하여 BUS에 직렬로 전송되어 모든 노드에서 수신 할 수 있다.

CAN 네트워크로 연결된 장치들은 일반적으로 센서, 액추에이터 등 기타 제어 장치들이다.
각 장치들(노드들)은 CAN Processor, CAN Controller, Can Transceiver를 통해 Bus에 연결된다.

## **데이터전송**

- CAN 데이터 전송은 **무손실 비트별 경합 해결 방법(lossless bitwise arbitration method of contention resolution)**을 사용한다.
	- CAN 네트워크 내의 모든 노드를 동기화하여 CAN 네트워크 내의 모든 비트를 동시에 샘플링해야 함(CAN 동기화라고 함)
	- 불행하게도, 데이터가 clock signal없이 비동기적으로 전송되기 때문에, 동기라는 용어는 부정확함.
- CAN specifications(이하 CAN spec)은 우성(dominant) 비트와 열성(recessive) 비트를 사용한다.
	- 우성 비트는 0, 열성 비트는 1
- Idle state는 열성(recessive) 레벨로 표현된다(logical 1)
- 만약 각각 서로 다른 두 노드가, 하나는 우성 비트를, 하나는 열성 비트를 보낸다면 우성 비트가 이긴다.
	- 우선순위가 높은 메시지(우성 메시지)의 경우 딜레이가 없다.
	- 우선순위가 낮은 노드가 메시지를 보내면 자동적으로 우성 메시지의 끝으로부터 6 bit clocks 뒤에 **재전송을 시도**한다.
	- 해당 방식은 실시간 우선 통신 방식에서 적합하다.
- 논리 0, 1의 정확한 전압은 물리 계층의 사용에 의존한다.
- CAN의 기본 원리에서는 각 노드가 전송 노드 자체를 포함하여 CAN 네트워크의 데이터를 수신해야 한다.
	- 논리 1이 모든 전송 노드에 의해 동시에 전송되면 전송 노드와 수신 노드를 모두 포함하는 모든 노드에 논리 1이 표시된다.
	- 반대의 경우(논리 0이 모든 전송 노드에 의해 동시에 전송될 때) 모든 노드에 0이 표시된다
	- 논리 0과 1이 각각 하나 이상의 노드에 의해 전송되는 경우, 1을 보낸 노드도 포함하여 모든 노드가 0이 표시된다.
		- 단일 노드가 논리 1을 전송했는데 0이 표시된다면, 경합이 발생하여 전송이 안됐다는 것을 알 수 있다.
		- 경쟁에서 진 노드는 다음 전송을 위해 큐에 메시지를 등록하고 CAN 프레임 비트스트림은 큐에 단 하나의 노드만 에러 없이 남을 때까지 계속된다.

## 파이썬을 통한 CAN 통신

기본적으로 canlib는 C언어로 구현되어있다. python canlib를 쓰기 위해서는 canlib.exe를 통해 CAN SDK를 설치해야한다.

```c
#include <canlib.h>
#include <stdio.h>
void main(void)
{
  canHandle hnd;
  canInitializeLibrary();
  hnd = canOpenChannel(0, canOPEN_EXCLUSIVE);
  if (hnd < 0) {
    char msg[64];
    canGetErrorText((canStatus)hnd, msg, sizeof(msg));
    fprintf(stderr, "canOpenChannel failed (%s)\n", msg);
    exit(1);
  }
  canSetBusParams(hnd, canBITRATE_250K, 0, 0, 0, 0, 0);
  canSetBusOutputControl(hnd, canDRIVER_NORMAL);
  canBusOn(hnd);
  canWrite(hnd, 123, "HELLO!", 6, 0);
  canWriteSync(hnd, 500);
  canBusOff(hnd);
  canClose(hnd);
}
```

1. CAN 통신을 위해서는 먼저 canInitializeLibrary()를 호출하여 드라이버를 초기화한다.
2. 초기화를 한 다음, canOpenChannel()을 통해 CAN 노드와 통신할 채널을 생성한다.

```python
from canlib import canlib, Frame

if __name__ == "__main__":
    try:
        ch = canlib.openChannel(
            channel=0,  # 연결할 노드의 채널 번호
            flags=canlib.Open.ACCEPT_VIRTUAL,  # 어떤 모드로 채널을 열 것인지 지정
            bitrate=canlib.Bitrate.BITRATE_250K,  # 비트레이트 지정
        )  # 지정된 flag로 채널 오브젝트가 만들어진다
    except canlib.exceptions.CanNotFound:
        print("canOpenChannel failed")
        exit(1)

    ch.setBusParams(canlib.Bitrate.BITRATE_250K, 0, 0, 0, 0, 0)
    ch.setBusOutputControl(canlib.Driver.NORMAL)
    ch.busOn()
    ch.write(Frame(id_=123, data=b"HELLO!", dlc=6))
    ch.writeSync(timeout=500)
    ch.busOff()
    ch.close()
```

1. 파이썬에서는 openChannel()를 호출하여 1,2번 과정을 동시에 수행한 뒤, 채널 객체를 생성한다.
2. setBusPrams() : 비트 타이밍을 설정한다
3. setBusOutputControl() : 드라이버 모드를 지정한다.
4. busOn() : CAN 버스 통신을 시작한다
	- 각 물리적 채널에 여러 개의 핸들을 사용할 수 있는데, 각 핸들마다 busOn()을 호출해야 한다.
	- busOff()에 대해서도 동일하게 적용된다
5. write() : CAN 메시지를 보낸다.
	- Frame 객체로 메시지를 보내며, 드라이버로 메시지를 큐에 넣은 뒤 바로 Return된다. → 즉, 바로 메시지가 전송되지는 않는다.
	- 같은 채널에 여러 핸들을 사용한다면, 각 핸들은 서로 "들을 수" 있다, e.g. A스레드 핸들에서 메시지를 전송하면, B스레드 핸들에서 해당 메시지를 받을 수 있다.
6. writeSync() : queue에 있는 메시지가 송신될 때까지 대기한다.
7. busOff() : CAN 버스 통신을 끝낸다.
	- 채널에 연결된 모든 핸들이 busOff가 되기 전까지는 채널은 bus에서 내리지 않는다.
8. close() : 채널을 닫는다.

### CAN Channel

- CAN 네트워크에서 노드와 통신할 수 있는 통로

### CAN Handle

- canOpenChannel 호출 시 Handle을 반환한다.
- 반환된 핸들은 이를 생성한 컨텍스트 안에서만 유효하다.
	- CAN BUS를 사용하는 애플리케이션의 각 스레드에서 canOpenChannel을 호출하여 각각 핸들을 생성해야 .
	- 즉, 한 채널에서 여러 핸들을 생성할 수 있다. 단, 각 스레드 마다 한 번씩 canOpenChannel을 호출해야 한다.

### CAN Channel Flag

Python canlib 기준 작성

- **EXCULSIVE :** 한 개 채널에 단 한 명만 접근할 수 있음
- **REQUIRE_EXTENDED :** 지정된 회로가 확장 CAN(CAN 2.0)을 허용하지 않으면 실패 또는 write 호출에 프레임 유형 플래그가 아닌 경우 확장 CAN을 사용해야 함
- **ACCEPT_VIRTUAL :** 가상 채널 여는 것을 허용
- **OVERRIDE_EXCLUSIVE :** 이미 열린 EXCLUSIVE채널이 있더라도 채널을 연다
- **REQUIRE_INIT_ACCESS : INIT_ACCESS**로 채널을 열 수 없으면 호출이 실패한다
	- **INIT_ACCESS**란 CAN 핸들러가 비트레이트와 드라이버 모드를 세팅할 수 있는 것.
	- 최대 한 개의 CAN 핸들러가 특정 채널에 대한 INIT ACCESS권한을 가질 수 있다.
	- 권한이 없는 핸들러가 설정을 하려고 하면 호출은 실패한다.(호출이 아무 동작하지 않아도 canOK가 반환 된다)
	- 기본값은 INIT_ACCESS이다.
- **NO_INIT_ACCESS :** INIT ACCESS로 핸들을 열지 않는다
	- 어떤 핸들도 채널을 init으로 열지 않는다면, NO_INIT_ACCESS로 열린 핸들은 여전히 기본 비트레이트를 설정 후 버스에 오른다.
- **ACCEPT_LARGE_DLC :** DLC 크기가 8보다 커질 수 있다.
- **CAN_FD :** CAN FD 프로토콜을 이용하여 채널을 연다.
- **CAN_FD_NONISO :** CAN FD NON_ISO 프로토콜을 이용하여 채널을 연다.

### CAN Driver Mode

setBusOutputControl()을 통해 Driver 타입을 설정한다.
CANlib는 직접적인 버스 출력 레지스터의 조작을 허용하지 않는다. 대신 심볼릭 상수를 통해 원하는 드라이버 모드를 선택할 수 있다.

- **NORMAL :** 기본 드라이버 타입
- **SILENT :** Silent 모드로 CAN Controller를 설정한다. 어떤 송신도 하지 않고, 수신만 한다.
- **SELFRECEPTION :** self-reception 모드. 구현되지 않음.
- **OFF :** 드라이버 모드를 끈다. 모든 하드웨어에서 구현되지 않음.

### Message Frame

파이썬에서 CAN통신 메시지를 보낼 때 Frame객체로 메시지를 송신한다.

**Args**

- id_ : 메시지 id
- data : 메시지 내용
	- dlc가 있는 경우, dlc와 일치하도록 남는 데이터는 0으로 채운다.
- dlc : 메시지 dlc(data lenght code) 메시지 길이. 기본적으로 메시지 길이로 계산된다.
- flags : 메시지 flag
- timestamp : Optional timestamp

### CAN Message Flag

CAN 메시지를 송수신할 때 메시지의 타입을 나타내는 flag

- **canMSG_RTR (0x0001) :** Message is a remote request.
- **canMSG_STD (0x0002) :** 11 bit 크기 id의 표준 메시지 형식
- **canMSG_EXT (0x0004) :** 29 bit 크기 id의 확장된 메시지 형식
- **canMSG_WAKEUP (0x0008) :** Message is a WAKEUP message (SWC hardware.)
- **canMSG_ERROR_FRAME (0x0020):** 에러 프레임을 나타내는 메시지
- **canMSG_SINGLE_SHOT (0x1000000) :** 재전송 없이 단 한 번(single-shot)만 메시지를 전송한다.

위 flag들은 CAN FD에서 사용 불가능하다.

### CAN Status

- **canOK :** 일반적인 성공 상태
- **canERR_PARAM :** 하나 이상의 파라미터에서 오류가 생김
	- 특정 파라미터가 유효하지 않을 때
		- out of range
		- invalid type
		- ...
	- 호출이 구현되지 않았을 때
- **canERR_NOMSG :** 읽을 메시지가 없을 때
- **canERR_NOTFOUND :** 특정 디바이스 또는 채널이 없을 때
	- 조건에 맞는 하드웨어가 없을 때
	- 이미 선점된 exclusive access채널에 요청할 때
