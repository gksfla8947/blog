# **기초**
### 무결성 체크
- Git은 데이터를 저장하기 전 SHA-1 해시를 이용해 체크섬을 만들어 무결성을 보장한다
\t- 체크섬은 `24b9da6552252987aa493b52f8696cd6d3b00373`와 같이 생겼다
### Git은 데이터를 추가할 뿐이다
Git Repository 관리에 있어서 중요한 사실이다.
Git은 Commit되기 전까지는 작업내용을 잃을 수 있지만, 한번 커밋이 되면, 파일이 삭제가 되든, 변경이 되든 데이터는 늘어나고 히스토리 역시 기록된다.
### 세 가지 영역
![](/images/notion/git-advanced/image-2.png)
세 가지의 상태와, 그에 따른 세 가지의 영역이 존재한다
- **modified, untracked**
\t- 아직 git 저장소에 반영되지 않은 수정된 내용 또는 추가된 파일, 삭제된 파일들이다
\t- staging area에 올라가 commit을 해야 비로소 해당 내용들이 반영되어 기록된다
- **staged**
\t- **modified, untracked **상태의 파일들이 staging area에 올라가 commit될 준비가 된 상태이다
\t- commit 시 staging area에 올라간 모든 내용이 git에 기록되게 된다
- **committed**
\t- staging area에 올라간 변경사항들이 기록되어 로컬 데이터베이스(.git)에 안전하게 저장되어 추적이 가능한 상태이다
# **Git Config**
이름 그대로 git의 동작을 결정하는 설정 정보들을 지정할 수 있는 도구이다
크게 **세 가지**의 Scope로 분류된다
- /etc/config : 시스템의 모든 사용자와 모든 저장소에 적용되는 설정
- \~/.gitconfig : 특정 사용자(즉 현재 사용자)에게만 적용되는 설정
- .git/config : Git 디렉토리에 있고 특정 저장소(혹은 현재 작업 중인 프로젝트)에만 적용
config 적용의 우선순위는 역순으로 적용된다. (directory → user → global)

> **참고**
\t**윈도우의 경우**
\tWindows에서는 `$HOME` 디렉토리에서 `.gitconfig` 파일을 찾는다(아마도 `C:\\Users\$USER` 디렉토리). Windows에서도 `/etc/gitconfig` 파일은 그 경로에서 찾는다.

### config 적용 방법
git config \[scope\] \[key\] \[value\]
```plain text
$ git config --global user.name "John Doe"
$ git config --global user.email johndoe@example.com
```
위와 같이, --global 또는 --system 또는 지정하지 않음으로써 scope설정이 가능하다
# **Log 조회**
기본적으로 `git log` 명령으로 조회가 가능하다
### 옵션
- p, --patch
\t- 각 커밋의 diff결과 보여준다
- -stat
\t- 각 커밋의 통계 정보를 보여준다
- -pretty=\[옵션\]
\t- oneline : 각 커밋 라인을 한 라인으로 축약해서 보여준다
\t- short : 일반 log에서 date를 축약
\t- full : 일반 log에서 committer를 추가
\t- fuller : full에서 commit date까지 추가
\t- format : 커스텀하게 포맷을 지정하여 출력하고 싶을 때 사용 
\t```shell
$ git log --pretty=format:"%h - %an, %ar : %s"
ca82a6d - Scott Chacon, 6 years ago : changed the version number
085bb3b - Scott Chacon, 6 years ago : removed unnecessary test
a11bef0 - Scott Chacon, 6 years ago : first commit
\t```

> **참고**
\t**Author와 Committer**
\t실제 파일을 작성 또는 수정한 자는 Author, 그러한 변경 사항을 git 저장소에 반영(commit)한 자를 committer라고 한다

- -since, --until
\t- 시간을 기준으로 조회가 가능하다
- -author
\t- author를 기준으로 조회가 가능하다
- -grep
\t- 특정 키워드에 매칭하는 커밋 메시지를 기준으로 조회가 가능하다
# **되돌리기**
git을 사용하다보면 잘못된 내용을 staging area에 올리거나 파일을 잘못 건들여 수정되는 일이 일어날 수 있다.
### 커밋 수정
먼저 되돌리기 전에 직전 커밋의 경우 수정이 가능하니 되돌리지 말고 아래의 방법을 사용한다
`git commit --amend`명령어로 직전에 했던 커밋 수정이 가능하다
1. 직전 커밋의 제목이 마음에 안들고 특정 파일 하나가 빠진 상황
\t![](/images/notion/git-advanced/image-3.png)
2. 먼저 직전 커밋에 반영할 내용을 staging해준다
\t![](/images/notion/git-advanced/image-4.png)
3. `git commit --amend`명령어를 입력해 편집기에서 commit 제목을 수정 후 저장
\t![](/images/notion/git-advanced/image-5.png)
4. 정상적으로 수정이 된 것을 볼 수 있다
\t![](/images/notion/git-advanced/image-6.png)

> **참고**
\t**REMOTE에 이미 반영된 커밋인 경우 반드시 amend한 커밋을 확인하자**
\tamend된 커밋은 제목만 바꿨다 하더라도 커밋의 **해시값**이 아예 바뀌기 때문에 Remote에 그대로 반영했다가는 충돌이 날 수 있다.
\t반드시 Remote와 Sync를 맞추고 잘못된 내용이 remote에 들어가지 않도록 조심하자
\t**보통 amend명령은 방금 전 커밋을 하고 바로 반영할 내용을 까먹었거나 제목이 마음에 안들 때(또는 오타) 즉시 수정용으로 사용을 한다.**

### Staging Area에서 내리기
만약 `git add *`명령어를 잘못 사용하여 staging area에 올라가지 말아야할 파일이 staged가 되었다면 아래 두 명령어로 다시 staging area에서 내릴 수 있다
- `git restore --staged [파일이름]`
- `git reset HEAD [파일이름]`
### 특정 파일을 직전 커밋의 내용으로 되돌리기
staging area에 올라가지 않은 특정 파일의 변경사항을 아예 취소하고 직전 커밋의 내용으로 되돌리고 싶다면 아래 명령어를 사용한다
- `git checkout -- [파일이름]`
# **Remote 저장소**
내가 git을 통해 작업한 결과물을 다른 사람들과 공유하기 위해서는 그냥 저장소 자체를 공유할 수도 있겠지만, 특정 원격지에 올라간 저장소에 내 로컬 git 저장소를 올려놓아 공유할 수 있다
### Remote 저장소 확인
`git remote`명령으로 확인할 수 있다.
기본적으로 Remote에서 Clone받은 로컬 저장소는 Remote 저장소의 이름을 **origin**으로 저장한다
![](/images/notion/git-advanced/image-7.png)
- v 옵션으로 Remote 저장소의 URL도 확인이 가능하다
![](/images/notion/git-advanced/image-8.png)
### Remote 저장소 추가
만약 내 로컬 저장소에 다른 Remote저장소의 특정 내용을 끌고 오고 싶다면 Remote 저장소에 추가해야 한다
`git remote add [별명] [Rmote URL]`명령으로 추가가 가능하다
![](/images/notion/git-advanced/image-9.png)
만약 추가한 Remote의 특정 브랜치를 가져오고 싶다면 먼저 remote 저장소를 fetch한뒤 (`git fetch [Remote별명])`
`git checkout -t [Remote 별명]/[Branch명]`을 실행하여 가져온다
![](/images/notion/git-advanced/image-10.png)
### Fetch와 Pull
- Fetch : 로컬과 대비하여 Remote 저장소의 최신 변경 내용을 로컬 저장소로 가져온다
\t- `git fetch [Remote]`로 fetch할 remote 저장소를 지정 가능하다
- Pull : Fetch뿐 아니라 Fetch한 내용을 대상 브랜치에 Merge까지 수행한다
\t- 즉, 해당 명령은 fetch - merge 두 단계를 수행하는 단축 명령어일 뿐이다
### Push
`git push [Remote명] [Push할 Branch명]`명령을 통해 로컬의 branch를 특정 Remote 저장소에 반영할 수 있다
![](/images/notion/git-advanced/image-11.png)
이렇게 Remote 저장소에 대한 이해와 명령어를 숙지하고 있다면
origin Remote 저장소 외에 제 3의 Remote 저장소의 내용을 Origin Remote 저장소에 반영하는 것이 가능하다

# **태그**
git의 태그는 보통 특정 커밋에 대한 형상에 버전 정보를 기입하기 위해 사용한다
기본적으로 `git tag`, `git tag --list` 명령으로 조회가 가능하다
- `git tag --list`의 경우 와일드카드를 통한 조회가 가능하다
\t![](/images/notion/git-advanced/image-12.png)
### Lightweight 태그와 Annotated 태그
- Lightweight 태그
\t- 단지 특정 커밋에 대한 포인터 역할만을 하는 태그이다
\t- 이름처럼 따로 tag에 대한 객체가 생성되지도 않고 그냥 커밋을 가리킬 뿐이다
\t- 별도 옵션 지정 없이 `git tag [tag명]`으로 생성가능하다
\t- `git show` 명령으로 확인 시 태그 정보 확인이 불가능 하다(이름만 보임)
\t\t![](/images/notion/git-advanced/image-13.png)
- Annotated 태그
\t- `git tag -a [tag명] -m [tag 설명]`으로 생성이 가능하다
\t- tag에 tag생성자 정보가 들어가며, git 객체로 관리가 된다
\t- `git show`명령으로 확인 시 태그 정보 확인이 가능하다
\t\t![](/images/notion/git-advanced/image-14.png)
### 태그 Push
태그의 Push는 `git push [Remote명] [태그명]`으로 가능하며, 태그만 Push하게 된다
- 만약 Remote에 없는 로컬의 모든 태그를 한번에 Push하고 싶다면 `git push --tags`를 사용한다
\t- 원치 않는 다른 태그들까지 모두 push하기 때문에 사용에 주의하도록 한다
# **브랜치**
브랜치는 git에서 가장 중요한 개념이자 git의 큰 장점 중 하나이다.
가장 중요한 것은 Branch는 특정 커밋을 가리키는 **포인터**이다. 사용자는 이러한 Branch를 자유자재로 넘나들며, 각 브랜치가 가리키는 커밋에 대한 형상으로 전환할 수 있다
### 기본적인 명령어
- 생성 : `git branch [branch명]`
- 이름 변경 : `git branch -m [branch명] [변경할branch명]`
- 삭제 : `git branch -d [branch명]`
\t- 만약 해당 branch가 바라보는 Remote 저장소의 브랜치가 있다면 `git branch -D [branch명]`으로 강제 삭제가 가능하다
### HEAD
HEAD는 git에서 특정 커밋 또는 브랜치를 가리키는 **포인터**이다.
![testing 브랜치를 바라보고 있는 HEAD객체](/images/notion/git-advanced/image-15.png)
기본적으로 branch를 바라보고 있을 때 커밋이 새롭게 생성되면, branch는 새로운 커밋으로 이동하고 자연스레 branch를 바라보는 HEAD역시 새로 생성한 커밋을 바라보게 된다
![](/images/notion/git-advanced/image-16.png)

> **참고**
\t**RefLog**
\tReflog는 HEAD가 가리키고 있었던 Branch, Commit, Tag 등 객체에 대한 히스토리이다.
\t![](/images/notion/git-advanced/image-17.png)


> **참고**
\t**HEAD Detached**
\t![](/images/notion/git-advanced/image-18.png)
\tHEAD가 branch를 가리키는 것이 아닌 Tag또는 특정 커밋을 가리킬 때 이러한 상태가 된다.
\t이 상태에서 커밋을 하나 생성 한다면, 새로운 커밋이 하나 쌓인 상태가 되고 새 커밋에 도달할 수 있는 방법이 따로 없게 된다.

### Checkout
브랜치의 이동은 `git checkout [Branch명]`으로 가능하다.
또한 몇 가지 사항들을 알아두면 좋은데, 아래와 같다
- Checkout은 Branch뿐만 아니라 Commit, Tag로도 이동이 가능하다
\t- HEAD객체가 가리키는 것을 바꾸는 명령어임
- `git checkout -b [Branch명]`입력 시 현재 HEAD에서 지정한 Branch명으로 Branch를 생성하여 동시에 이동하는 것이 가능하다
- `git checkout -t [Remote명]/[Branch명]`추적중인 Remote저장소의 특정 브랜치를 가져와 Local에 같은 이름으로 생성할 수 있다.
\t- 생성된 브랜치는 자동으로 Remote저장소의 브랜치를 추적하게 된다
\t\t![](/images/notion/git-advanced/image-19.png)
### Remote Branch
로컬에서 브랜치를 생성하면 기본적으로 어떠한 Remote를 가리키지 않는다. 따라서 Pull을 하기 위해서는 Remote Branch 지정이 필요하다
![](/images/notion/git-advanced/image-20.png)
# **Merge**
Merge는 특정 브랜치의 변경사항을 대상 브랜치에 합쳐서 반영하는 것이다
`git merge [합칠Branch명]`명령으로 수행할 수 있으며, 기본적으로 Fast-Forward가 가능하다면 Fast-Forward 방식으로 Merge를 수행, 안된다면 3-way Merge 방식으로 Merge를 수행한다
### 3-way Merge
Merge Commit을 남기는 방식의 Merge 방식이다.
- Dest Branch의 커밋, Source Branch의 커밋 그리고 Dest Branch에서 합쳐질 커밋 총 세 개의 커밋을 통해 merge를 수행하기 때문에 3-way Merge라고 불린다
![](/images/notion/git-advanced/image-21.png)
![](/images/notion/git-advanced/image-22.png)
### Fast-Forward
Merge Commit을 생성하지 않고 Dest Branch가 가리키는 커밋을 Source Branch의 최신 커밋으로 옮긴다
![](/images/notion/git-advanced/image-23.png)
![](/images/notion/git-advanced/image-24.png)

> **참고**
\t**Dest Branch에 Fast-Forward가 불가능한 상황**
\t만약 Dest Branch에 Source Branch에 없는 커밋이 존재하여 머지가 불가능한 상황이라면 Rebase를 통해 깔끔하게 머지를 할 수 있다.

### Remote Branch와 Pull의 동작
git clone 시에 기본적으로 로컬 저장소는 Remote 저장소의 이름을 **origin**으로 저장한다.
branch가 바라보고 있는 Remote의 정보를 확인하려면 `git branch -vv`명령으로 확인할 수 있다
![](/images/notion/git-advanced/image-25.png)
유용한 명령어로는
- `git push --set-upstream [Remote명] [Branch명]`: Remote 저장소에 없는 로컬 브랜치를 Remote에 Push
- `git branch --set-upstream-to=[Remote Branch명]`: 로컬 브랜치가 가리키는 Remote 브랜치를 변경 또는 추가
- `git branch -b [Branch명] [연결할 Remote Branch명]`: 로컬에서 브랜치를 생성하면서 Remote 브랜치를 연결
\t- 별도 이름 지정없이 Remote Branch명과 똑같이 하고 싶다면 `git checkout -t [Remote명]/[Branch명]`
\t\t- 만약 Local에 Remote Branch명의 Local Branch가 없다면 -t 생략이 가능하다
**Pull의 동작**
1. Clone을 통해서든 --set-upstream-to 옵션을 통한 remote branch 지정이든 먼저 로컬 브랜치가 가리키는 Remote 브랜치가 있어야 한다
\t![](/images/notion/git-advanced/image-26.png)
2. Remote Branch의 커밋이 업데이트되어 변경된다. 그 사이 로컬에서는 Remote에서 변경되기 전 커밋을 기준으로 또 다르게 업데이트 된다
\t![](/images/notion/git-advanced/image-27.png)
3. `git fetch [Remote명]`명령을 통해 Remote의 업데이트를 가져온다
\t![](/images/notion/git-advanced/image-28.png)
4. Local로 가져온 Remote 브랜치의 내용을 Local 브랜치로 Merge한다
\t![](/images/notion/git-advanced/image-29.png)
`git pull origin main`명령은 위 4개의 일련의 동작을 순서대로 수행한 것이다
# **Rebase**
아래와 같이 Branch를 Merge하는 상황에서 Merge commit을 남기지 않고 Merge를 어떻게 할까?
![](/images/notion/git-advanced/image-30.png)
Rebase를 통해 Merge Commit없이 Fast-Forward가 가능하다
위 예제의 경우 아래 순서를 통해 Rebase 후 Merge가 가능하다
1. Merge될 Source Branch로 Checkout
\t![](/images/notion/git-advanced/image-31.png)
2. `git rebase [DestBranch명]`실행
\t![](/images/notion/git-advanced/image-32.png)
3. 다시 Dest Branch로 Checkout
\t![](/images/notion/git-advanced/image-33.png)
4. merge 수행 - `git merge feature`
\t![](/images/notion/git-advanced/image-34.png)

> **참고**
\t**Remote에서 Pull 수행 시 Rebase가 필요한 경우**
\t아래와 같은 상황도 Rebase를 수행함으로써 커밋이 깔끔하게 정리가 된다
\t![](/images/notion/git-advanced/image-35.png)
\t일반적으로 Rebase수행하지 않고 Pull 수행 시 아래와 같아진다
\t![](/images/notion/git-advanced/image-36.png)
\t하지만 `git pull --rebase`명령어로 pull을 한다면 아래처럼 깔끔하게 Rebase 후 Merge가 수행된다
\t![](/images/notion/git-advanced/image-37.png)

# **Stash & Clean**
작업을 하고있던 도중 다른 Branch로의 이동이 필요할 때(Checkout) 아직 완성도 안된 결과물을 commit하거나 또는 그대로 옮겨갈 Branch로 가져오기 껄끄러울 때가 있다
Stash를 통해 작업하고있는 내용을 임시로 저장한 뒤 나중에 다시 꺼내올 수 있다
- `git stash`명령을 통해 staging area에 올라가지 않은 변경사항을 보관할 수 있다
\t- 만약 staging area의 변경사항도 stash하고 싶다면 `S, --staged`옵션을 사용한다
- `git stash pop`: stash된 수정사항을 다시 working directory에 반영
- `git stash clear`: stash된 수정사항을 전부 clear
- `git stash drop {번호}`: 특정 stash index의 수정사항을 삭제
\t- stash는 stack처럼 수정사항이 쌓여서 보관된다
빌드 후 생겨난 파일들이나 더럽게 쌓인 것들을 전부 지우고싶을 때는 `git clean`명령을 사용한다
# **RESET**
reset 명령은 HEAD를 옮기는 것이 아니라 Branch가 바라보는 commit자체를 옮긴다는 점에서 checkout과 대조된다
**working directory, staging area, committed **세 개의 area의 수정사항이 어떻게 바뀌냐에 따라 **--soft, --mixed, --hard **reset의 동작이 다르다
### git reset --soft
![](/images/notion/git-advanced/image-38.png)
HEAD가 가리키는 Branch의 최신 커밋을 옮기면서, Staging Area와 Working Directory는 이전 내용이 그대로 적용되어 있다.
즉, staging area에 올라가 커밋되기 전 상태로 돌아간 것이다
![](/images/notion/git-advanced/image-39.png)
### git reset --mixed
![](/images/notion/git-advanced/image-40.png)
working directory를 제외하고 staging area까지 이전 커밋 내용으로 되돌려놓는다
![](/images/notion/git-advanced/image-41.png)
### git reset --hard
![](/images/notion/git-advanced/image-42.png)
working directory를 포함하여 모든 내용이 이전 커밋으로 돌아간다.
**돌아가기 전 커밋이 저장되 곳이 별도로 없다면 복구하는 것을 불가능하다**
![](/images/notion/git-advanced/image-43.png)
### Reset을 통한 Squash(커밋 합치기)
아래 사진의 log의 상위 세 개의 커밋을 합칠 것이다
![](/images/notion/git-advanced/image-44.png)
1. 세 개 커밋 전으로 soft reset
\t![](/images/notion/git-advanced/image-45.png)
2. 그대로 staging area에 있는 변경사항을 commit한다
\t![](/images/notion/git-advanced/image-46.png)
# **Submodule**
프로젝트 수행 시 다른 프로젝트를 사용하는 경우 Git 저장소 안에 다른 Git 저장소를 디렉토리로 분리해 넣어 서브모듈로 사용한다.
### Submodule 생성
메인 레포지토리 클론 : submodule을 추가할 메인 레포지토리를 clone한다.
\$ git clone \[URL\] // URL은 상대경로, 절대경로 둘 다 가능하다.
\$ git status
서브 모듈 추가한 뒤 .gitmodules파일이 생성되는데, 이 파일은 서브디렉토리와 하위프로젝트 url을 매핑한 설정파일이 있다.
